# 后台管理页响应式与布局修复设计

日期：2026-06-27

## 背景

当前后台管理页已经切到 card + table 形态，但在实际使用中存在三个直接问题：

1. 域名列表区域无法正常滚动，尤其在窄屏下容易被外层布局卡死。
2. 搜索、新增、筛选等表单控件布局松散，视觉上偏“居中堆叠”，缺少统一对齐。
3. 页面只对桌面宽屏友好，手机与小屏笔记本下导航、表单、表格都缺少可靠响应式策略。

本次仅调整后台已登录页面中的三个页面：

- `/` 域名页
- `/upstreams` 回源页
- `/config` 配置页

不调整登录页，不修改后端接口语义，不引入新的前端运行时依赖。

## 目标

- 修复域名页和回源页表格容器的滚动问题，保证手机和桌面端都可访问完整列表。
- 统一域名页和回源页表单布局，使输入区、筛选区、按钮区在桌面端左对齐，在手机端顺序下沉。
- 将后台主框架改为真正的响应式导航：
  - 桌面端保留左侧导航 + 顶栏
  - 手机端改为顶栏 + 抽屉菜单
- 将配置页改为稳定的响应式卡片布局，解决长文本撑破和键值不对齐问题。

## 非目标

- 不重做视觉品牌风格，不更换组件库。
- 不把表格在手机端改造成另一套独立的卡片列表。
- 不新增主题切换、暗色模式、复杂动效或可折叠菜单状态持久化。
- 不修改 domains/upstreams 的业务交互流程、批量逻辑和后端路由。

## 设计决策

### 1. 主布局改为双形态导航

当前 `layout.html` 在所有已登录页面固定渲染左侧栏，这会直接压缩手机视口。新布局采用两套显示策略：

- `lg` 及以上：保留当前左侧栏，宽度固定，主内容区使用 `min-w-0` 防止被内部子元素撑出。
- `lg` 以下：隐藏左侧栏，在顶栏增加菜单按钮，通过 daisyUI `drawer` 展开导航。

这样可以保留桌面端信息密度，同时让手机端优先显示内容区。导航仍保持三项：域名、回源、配置。

### 2. 列表页容器层级收敛，滚动只放在表格卡片内部

列表不可滚动的核心风险不是“表格太宽”，而是多个祖先容器同时参与宽度裁剪，导致 `overflow-x-auto` 失效。修复策略：

- 主内容区统一 `w-full min-w-0`
- 页面根节点统一 `w-full`
- 列表卡片统一 `overflow-hidden`
- 真正负责横向滚动的元素只保留一层：包裹 `<table>` 的 `overflow-x-auto`
- 表格本身增加合理最小宽度，避免列被过度压缩

结果是：

- 桌面端保持正常表格布局
- 小屏和手机端允许左右滑动查看完整列
- 不依赖写死高度或脚本强行计算尺寸

### 3. 搜索与新增区统一为响应式栅格

域名页和回源页目前把搜索、筛选、新增表单混在一行里，且靠 `ml-auto` 推动按钮组，导致宽度变化时容易“中间一坨”。统一改法：

- 卡片 body 改为响应式纵向堆叠基础结构
- 搜索区独占一行或一列，始终左对齐并拉满可用宽度
- 筛选器放入固定宽度但可换行的区块
- 新增表单独立成响应式 grid：
  - 手机端单列，按钮全宽
  - 平板端两列或多列
  - 桌面端保持同行且字段边缘对齐
- 批量导入按钮跟随新增表单区域，不再漂到内容正中

域名页和回源页会使用同一套布局类结构，仅字段数量不同。

### 4. 配置页改为稳定卡片网格与自适应键值布局

配置页继续保持五个分组，但对布局进行约束：

- 页面根容器使用单列到双列、再到宽屏多列的渐进式 grid
- 所有卡片内容使用统一的 `dl` 样式
- 小屏下 `dt/dd` 改为单列堆叠，避免标签列挤压值内容
- 中屏及以上恢复两列键值结构
- 路径、URL、命令等长文本使用断行或 `break-all`，防止卡片被撑爆

这样配置页在手机上可读，在桌面上仍保留信息密度。

### 5. 不单独改写手机专属模板

可选方案是给列表页在手机端切为卡片列表，但这会让模板维护成本翻倍，而且这次用户诉求主要是“手机可用”，不是“手机最佳”。因此保留单套表格模板，仅通过导航和滚动策略实现响应式。

## 受影响文件

- `internal/web/template/layout.html`
- `internal/web/template/domains.html`
- `internal/web/template/upstreams.html`
- `internal/web/template/config.html`
- `web/input.css`
- `internal/web/static/tailwind.css`

如现有测试不足，还会补充后台页面渲染测试，至少校验关键布局结构存在。

## 测试策略

先补失败测试，再实现：

1. 页面渲染测试覆盖关键结构类名或标记：
   - 移动端抽屉按钮/导航结构
   - 列表滚动容器
   - 响应式表单容器
   - 配置页响应式网格结构
2. 执行相关 Go 测试，确认模板渲染未回归。
3. 重新生成 `internal/web/static/tailwind.css`。
4. 进行一次最终回归，确保不存在缺失类名导致的样式失效。

## 风险与缓解

- 风险：导航结构修改可能影响所有已登录页布局。
  缓解：导航只改模板结构，不改路由和鉴权逻辑；通过页面渲染测试覆盖三页。

- 风险：Tailwind 新类名未进入产物，导致本地模板已改但页面无样式。
  缓解：修改后立即重新生成 `tailwind.css` 并做 diff 检查。

- 风险：表单对齐优化后，HTMX 嵌套 form 行为意外受影响。
  缓解：不改动现有请求目标和触发器，只调整容器与类名；保留现有表单边界。

- 风险：手机端导航抽屉引入后，主内容可用宽度仍被某个祖先容器限制。
  缓解：统一检查 `flex-1`、`min-w-0`、`overflow-x-auto` 的层级关系，并以列表页作为验证基准。

## 实施顺序

1. 为后台页面补充最小渲染回归测试，先让新结构断言失败。
2. 修改 `layout.html` 实现桌面/手机双形态导航。
3. 修改 `domains.html` 和 `upstreams.html`，统一搜索/新增/列表容器结构。
4. 修改 `config.html`，实现响应式卡片和键值布局。
5. 更新 Tailwind 输入样式与构建产物。
6. 执行测试与样式构建验证。
